# Viva Notes

## 1-Minute Project Introduction

My project is a Smart Meter Anomaly Detection Dashboard. It monitors smart meter readings in real time and detects abnormal behavior using two anomaly detection algorithms: `IsolationForest` and `LSTM`. Historical readings are used as the normal baseline, live data is generated by a simulator acting as a monitoring agent, and the backend analyzes each reading before showing the result on a dashboard. The system can highlight theft-like and fault-like behavior, show alert-related outputs, and support testing using simulated abnormal spikes.

## Short Self Introduction

I built this project as an academic demo of how AI can be integrated into smart meter monitoring. It combines historical data preparation, time-series preprocessing, anomaly detection, live telemetry, dashboard visualization, and evaluation in one system.

## Core Viva Questions And Answers

### 1. What is the aim of the project?

The aim is to monitor smart meter readings continuously and detect abnormal power behavior that may indicate electricity theft or faults.

### 2. Why did you choose anomaly detection?

Because abnormal meter behavior is not always predefined. Anomaly detection helps identify suspicious readings even when exact fault patterns are not hard-coded in advance.

### 3. What historical data does the project use?

The project uses `sample_data.csv` as the historical baseline dataset of normal smart meter readings.

### 4. Why is preprocessing important for this project?

Preprocessing prepares the time-series data so the models can learn meaningful patterns. It improves stability and makes both `IsolationForest` and `LSTM` usable for runtime detection.

### 5. What preprocessing steps are used?

The project uses:

- feature selection
- baseline preparation
- scaling for `IsolationForest`
- sequence window creation for `LSTM`
- threshold learning from normal behavior

### 6. Which machine learning models are used?

I used:

- `IsolationForest`
- `LSTM`

### 7. Why did you use Isolation Forest?

Isolation Forest is good for point anomaly detection. It quickly identifies unusual individual readings compared with normal baseline behavior.

### 8. Why did you use LSTM?

LSTM is good for sequence data. It checks whether the current power pattern fits the recent sequence of readings from the same meter.

### 9. Is this supervised or unsupervised learning?

It is mainly unsupervised or semi-unsupervised anomaly detection because the models learn normal behavior first and then identify unusual patterns.

### 10. What are the input features?

The input features are:

- `voltage`
- `current`
- `power`

### 11. What does the backend predict?

The backend produces:

- `prediction`: `NORMAL` or `ANOMALY`
- `detected_event`: `NORMAL`, `THEFT`, `FAULT`, or `ANOMALY`
- `confidence`
- `anomaly_source`

### 12. How is theft-like behavior detected?

If a reading is anomalous and the power is abnormally low, the system classifies it as `THEFT`.

### 13. How is fault-like behavior detected?

If a reading is anomalous and the power is abnormally high, the system classifies it as `FAULT`.

### 14. What is the role of the live monitoring agent?

The live monitoring agent generates or collects meter readings and sends them to the backend. In this repository, `meter_simulator.py` performs that role.

### 15. What does the dashboard show?

The dashboard shows:

- total visible readings
- anomaly counts
- theft-like and fault-like detections
- recent events
- live power graph
- confidence values
- algorithm-level results

### 16. What is an automated alert trigger in this project?

An automated alert trigger is the backend condition that marks a reading as anomalous when `IsolationForest` or strong `LSTM` sequence logic indicates suspicious behavior.

### 17. How do you test the project?

I test it using simulated abnormal spikes. The simulator injects low and high abnormal values so I can observe whether the system detects them correctly.

### 18. What is a false positive?

A false positive is when the system marks a normal reading as anomalous.

### 19. What is precision?

Precision means how many of the detected anomalies were actually true anomalies.

### 20. How do you evaluate the system?

The simulator provides labels, and the backend returns detected events. By comparing them, I can observe matches, wrong detections, false positives, and general detection quality.

### 21. What is confidence in this project?

Confidence shows how strongly the current anomaly logic supports the decision based on model outputs.

### 22. Why does the LSTM sometimes show warmup?

The LSTM needs a short history of previous readings for a meter before it can evaluate a full sequence. Until then, it stays in warmup mode.

### 23. What technologies are used?

The main technologies are:

- Python
- FastAPI
- Flask
- Chart.js
- scikit-learn
- NumPy
- pandas
- requests

### 24. How is this project deployed?

The current version runs as a local direct-mode demo, where the simulator sends readings directly to the backend and the dashboard visualizes the results.

### 25. What is the AI workflow in this project?

The AI workflow is:

1. collect historical smart meter data
2. preprocess the time-series data
3. train anomaly detection models
4. receive live readings from monitoring agents
5. run inference in the backend
6. visualize anomalies on the dashboard
7. trigger anomaly alerts
8. evaluate false positives and matches
9. improve the model and thresholds

### 26. What is monitoring integration?

Monitoring integration means the AI model is connected to the full runtime pipeline. It receives live data, produces anomaly decisions, supports alert logic, and updates the dashboard continuously.
